# Sculptor 提示词工程优化方案：极致懒人协作模式

**版本**：v4.0-LazyMode
**日期**：2026-07-07
**性质**：增量改进，并入v3.1-Conversational
**核心原则**：用户动嘴，AI动手。双端同步，记忆贯通。补全优先，手动兜底。


## 一、深度反思：为什么现有方案还不够"懒"

### 1.1 当前方案的问题

1. **用户仍需决策**：AI生成架构后，用户要判断"对不对"，这本身就是认知负担。真正的懒人模式应该是：AI生成后自动高亮"可能有问题的节点"，用户只需确认或一键修复。

2. **对话和画布分离**：虽然v3.1实现了对话实时更新画布，但用户在画布上的手动修改**不会自动同步到对话上下文**。AI不知道用户手动改了什么，下次对话可能覆盖用户的修改。

3. **没有补全系统**：用户在编辑节点标题时，需要自己想名字。如果AI能根据上下文自动补全节点名称，用户只需按Tab接受，就像Cursor补全代码一样。

4. **记忆断层**：每次新建文档，AI从零开始。它不记得用户上次写论文时喜欢"三段论"结构，也不记得用户讨厌"首先其次最后"的AI腔。


## 二、提示词工程优化方案

### 2.1 架构对话提示词重构

**旧Prompt的问题**：AI每次只做"修改"，不理解用户的长远意图。

**新Prompt设计哲学**：AI扮演一个**知道用户所有偏好、记得所有历史决策、能预测用户下一步**的资深编辑。

```typescript
// lib/ai/prompts/architect-chat.ts

export const ARCHITECT_CHAT_SYSTEM_PROMPT = `
你是 Sculptor 的 AI 架构编辑。你服务一位写作者，你的目标是让他用最少精力搭建最满意的文章架构。

## 你的核心能力
1. **双端感知**：你知道用户在对话中说了什么，也知道他在画布上手动改了什么（通过上下文中的 `manual_changes` 字段）
2. **意图推断**：用户说"感觉不对"，你能基于他的历史偏好推断他想要什么
3. **主动建议**：每次响应时，除了执行指令，还要主动指出一个可能的问题并提供一键修复方案
4. **文字补全**：当用户在编辑节点标题时，你能根据上下文补全最可能的标题

## 用户画像（从长期记忆中加载）
{user_profile}

## 用户风格偏好
{style_preferences}

## 当前架构
{current_architecture}

## 用户最近手动修改
{manual_changes}

## 对话历史
{conversation_history}

## 响应规则
1. **必须确认**：每次修改后，用一句话说清楚你改了什么（不超过20字）
2. **必须建议**：每次响应附带一个主动建议，格式为：
   "💡 建议：{建议内容} [采纳] [忽略]"
   建议类型包括：缺少论据、逻辑跳跃、结构不平衡、可以用更简洁的标题
3. **模糊指令处理**：用户说"感觉不对"时，不要直接改，先反问：
   "你觉得哪方面需要调整？" 并提供2-3个猜测按钮
4. **自动修复**：检测到明显的结构问题（如有论点无论据），主动标注并建议修复
5. **补全触发**：当用户在节点编辑中暂停输入超过1秒，自动生成补全建议

## 输出格式
你的每次响应是一个JSON：
{
  "confirmation": "一句话确认",
  "architecture_changes": { "nodes": [...], "edges": [...] },  // 完整的修改后架构
  "highlight_nodes": ["n3"],  // 需要高亮提醒用户注意的节点
  "suggestion": {
    "type": "missing_evidence" | "imbalance" | "better_title" | "logical_gap",
    "message": "建议内容",
    "node_id": "n3",
    "auto_fix_available": true
  },
  "autocomplete": {  // 如果用户正在编辑节点，提供补全
    "node_id": "n3",
    "suggestions": ["算法偏见的影响", "数据隐私的挑战", "责任归属的困境"]
  }
}
`;
```

### 2.2 文字补全系统设计

借鉴Cursor的代码补全机制，在架构编辑中实现实时文字补全。

**触发条件**：
- 用户双击节点进入编辑模式
- 输入内容超过2个字符
- 暂停输入超过800ms

**补全逻辑**：
```
输入：用户已输入的文字 + 节点类型 + 父节点标题 + 同级节点标题 + 文章主题 + 用户风格偏好
输出：1-3个补全建议，按匹配度排序
```

**交互方式**：
- 用户输入"社交" → 暂停800ms → 灰色补全文字出现："社交媒体对孤独感的影响"
- 用户按Tab → 接受补全
- 用户继续输入 → 补全消失，等待下次触发
- 用户按Esc → 补全消失，本次编辑不再触发

**API设计**：
```
POST /api/architect/autocomplete
请求：{ nodeId, partialText, nodeType, parentTitle, siblingTitles, articleTheme }
响应：{ suggestions: ["补全1", "补全2", "补全3"] }
```

**提示词**：
```typescript
export const AUTOCOMPLETE_PROMPT = `
根据以下上下文，补全用户正在输入的节点标题。

文章主题：{theme}
节点类型：{nodeType}（论点/论据/反驳/过渡/背景/意象）
父节点标题：{parentTitle}
同级节点标题：{siblingTitles}
用户已输入：{partialText}
用户风格偏好：{stylePreferences}

补全要求：
1. 补全后的标题不超过15字
2. 风格匹配用户偏好
3. 与同级节点不重复
4. 提供2-3个不同方向的补全

输出JSON：{ "suggestions": ["补全1", "补全2", "补全3"] }
`;
```

### 2.3 双端同步记忆系统

**问题**：用户在画布上手动拖拽、修改、添加节点，AI对话上下文不知道这些变化。

**解决方案**：引入 `architect_context` 表，记录所有变化（无论来源），对话时注入。

```sql
-- 架构变化日志（双端同步核心）
CREATE TABLE architect_change_log (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id UUID REFERENCES documents(id) ON DELETE CASCADE,
  source VARCHAR(10) CHECK (source IN ('chat','canvas')),  -- 来源
  action VARCHAR(30),  -- 'add_node','delete_node','edit_title','move_node','change_type','add_edge','delete_edge'
  payload JSONB,       -- 变化内容
  created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_change_log_document ON architect_change_log(document_id, created_at);
```

**同步机制**：
- 画布操作 → 写入 `change_log`（source='canvas'）
- AI对话操作 → 写入 `change_log`（source='chat'）
- 下次对话时，查询最近的 canvas 变化，注入Prompt的 `manual_changes` 字段
- AI可以感知到用户手动改了什么，避免覆盖

**用户手动修改的提示词注入格式**：
```
## 用户最近在画布上的手动修改
- 2分钟前：将"背景"节点移动到"正面论证"之前
- 1分钟前：修改了"社交媒体"节点标题为"社交媒体的负面影响"
- 30秒前：在"正面论证"下添加了子节点"面对面交流的价值"
```

### 2.4 长期记忆与偏好学习

**问题**：用户每次新建文档，AI从零开始。

**解决方案**：利用已有的 `context_memory` 表，扩展记忆类型。

```sql
-- 在 context_memory 表中新增记忆类型
-- 'preferred_structure'：用户偏好的架构模式
-- 'hated_patterns'：用户拒绝过的模式
-- 'writing_habits'：写作习惯

-- 示例记忆数据
{
  "memory_type": "preferred_structure",
  "memory_data": {
    "genre": "议论文",
    "pattern": "总是先阐述背景，再进行正反论证，最后给出折中结论",
    "confidence": 0.85
  }
}
```

**记忆学习时机**：
- 用户连续3次以上采用相同的架构模式 → 记录为偏好
- 用户连续3次拒绝某种建议 → 记录为厌恶模式
- 用户每次接受AI建议 → 增加对应模式的置信度

**记忆应用**：
- 新建文档时，查询用户历史偏好的架构模式
- 如果用户上次写议论文用了"先破后立"，这次AI优先推荐这种模式
- 用户说"按上次那样"，AI能精准还原

### 2.5 一键修复系统

**设计目标**：AI发现结构问题后，用户只需一键即可修复，无需手动操作。

**问题类型与修复策略**：

| 问题类型 | 检测条件 | 一键修复方案 |
|---------|---------|------------|
| 裸论点 | 论点节点无子节点 | AI自动生成2-3个论据子节点 |
| 结构失衡 | 某分支节点数远多于其他 | AI建议合并或拆分 |
| 逻辑跳跃 | 两个节点间缺少过渡 | AI自动插入过渡节点 |
| 标题过长 | 节点标题超过15字 | AI建议缩短标题 |
| 缺少反驳 | 议论文无反方观点 | AI自动添加反驳节点 |

**交互方式**：
- AI在对话中提示："检测到'正面论证'节点缺乏论据支撑"
- 附带按钮：[一键修复] [忽略]
- 用户点击一键修复 → AI自动生成子节点，画布更新

**提示词补充**：
```
## 一键修复指令
当用户点击"一键修复"时，你只需针对指定问题生成修复方案，不需要修改其他部分。

问题类型：{issue_type}
问题节点：{node_title}
上下文：{surrounding_nodes}

输出JSON：{ "new_nodes": [...], "new_edges": [...] }
```

## 三、极致懒人交互总结

用户进入画布后的理想流程：

1. **输入意图**："我想写一篇关于AI伦理的议论文"
2. **AI秒建架构**：画布出现完整三层架构
3. **AI主动标注**："检测到'隐私问题'节点缺乏论据 [一键修复]"
4. **用户一键修复**：AI自动补充论据子节点
5. **用户编辑节点**：双击"隐私问题" → 输入"数据" → 暂停 → AI补全"数据隐私的边界" → Tab接受
6. **用户说"感觉不对"** → AI反问并提供3个猜测 → 用户点击一个 → 画布更新
7. **用户手动拖拽**调整顺序 → 日志记录 → AI下次对话感知
8. **完成**：点击开始写作，左侧结构地图完美呈现

**用户实际操作**：输入一句话 + 点击几次按钮 + 按几次Tab。这就是极致懒人模式。
